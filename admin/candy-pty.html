🖥️ CandyPty — Operations

Operational guide for embedded PTY usage

Signals handled

  • SIGWINCH — When a terminal resize is detected via SignalForwarder::attachSigwinch(), the PTY is resized via ioctl(TIOCSWINSZ). If ext-pcntl is present, the forwarder uses async signals; otherwise it degrades to a no-op and the host terminal resizes without being propagated to the child.
  • SIGCHLD — The SignalForwarder also attaches a SIGCHLD handler to reap zombie children. If ext-pcntl is absent, waitpid() is polled manually in the pump loop instead.

Files written

  • No persistent files. Candy-pty is a pure in-memory library. It writes nothing to disk unless you explicitly call Recorder::open() to capture a session to a .cas file. That file is owned by the application, not by the library.

Child-exit detection

Child::exited() and Child::wait() first attempt a waitpid(pid, &status, WNOHANG) call via FFI — a sub-ms non-blocking probe. If the child has exited, the exit code is captured immediately. If FFI is unavailable or waitpid is not in the cdef, the methods fall back to PHP's proc_get_status() with a 10 ms poll interval.

Signal-terminated exits use the Unix convention: exit code is 128 + signal_number (e.g. SIGTERM=15 → exit code 143). This matches shell convention and is compatible with bash -c 'trap ... TERM; kill -TERM $child' patterns.

Always call $child->wait() or poll $child->exited() to reap the child — unreaped children become zombies.

Controlling terminal claim mechanism

When controllingTerminal: true is passed to spawn(), the child process claims the slave PTY as its controlling terminal via ControllingTerminal::claim() — a two-step FFI sequence:

  1. setsid() — creates a new session with the child as session leader (required before the kernel honours TIOCSCTTY).
  2. ioctl(fd, TIOCSCTTY, 0) — assigns the fd as the session's controlling terminal. The third arg 0 means "don't steal an existing ctty from another session." Platform-specific request codes: Linux 0x540E, macOS 0x20007461.

Once claimed, the kernel delivers job-control signals to the child: Ctrl+C → SIGINT, host resize → SIGWINCH, master close → SIGHUP. The implementation is in ControllingTerminal::claim() and is invoked by bin/pty-shim.php for the controllingTerminal: true spawn path.

The shim adds ~5–50 ms of startup latency per spawn (PHP boot + FFI cdef). Only enable controllingTerminal: true for interactive processes that need job control.

Platform behaviour — PTY allocation

  • macOS — openpty(3) is attempted first (single call that opens both master and slave fds). On failure it falls back to the POSIX quartet: posix_openpt + grantpt + unlockpt + ptsname_r.
  • Linux — Uses the POSIX quartet directly. The openpty(3) symbol lives in libutil.so.1 on most distributions — a separate FFI handle is opened specifically for it. If libutil.so.1 is unavailable, the quartet is used instead.

This distinction matters for debugging: if macOS PTY allocation fails, check whether openpty or the quartet path was taken (enable FFI tracing or check the return code path in PosixPtySystem::openPtyMaster()).

Failure modes

  • UnsupportedPlatformException — Thrown by PtySystemFactory::default() on Windows (v1 is POSIX-only), or when SUGARCRAFT_PTY_BACKEND is set to sidecar or pecl (not yet implemented).
  • InvalidArgumentException — Thrown when SUGARCRAFT_PTY_BACKEND is set to an unrecognised value.
  • FFI load failure — If ext-ffi is not available, or the libc path is wrong, the FFI cdef() call throws. Ensure ext-ffi is loaded and SUGARCRAFT_LIBC points to a valid file on non-standard systems.
  • Child deadlock — If a child process writes more than the kernel pipe buffer (~64 KiB) and the parent is not reading, the child blocks on write and the parent deadlocks waiting for the child to exit. Use non-blocking reads (setBlocking(false)) or PosixPump to avoid this.
  • Zombie children — Every spawn() must be followed by $child->wait() or $child->exited() polling to reap the exit status. Without reaping, the child becomes a zombie.

Resource cleanup

Always call $pty->close() when done. The method is idempotent — calling it multiple times is safe. If you used controllingTerminal: true, the shim process (bin/pty-shim.php) is reaped by the same $child->wait() call.
