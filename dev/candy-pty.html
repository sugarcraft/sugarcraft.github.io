Extending and testing the PTY backend

Extension points

The primary extension point is the Contract\PtySystem interface. Implement this interface to provide a custom PTY backend (e.g. Windows ConPTY sidecar, pecl extension):

interface PtySystem
{
    public function open(int $cols, int $rows): PtyPair;
}

Where PtyPair, MasterPty, SlavePty, and Child are also contracts in Contract/. See candy-pty/src/Contract/ for the full namespace tree.

Backend conventions

All backends must be selectable via SUGARCRAFT_PTY_BACKEND. The factory PtySystemFactory::default() reads this env var and dispatches:

ValueBackend classStatus
posix-ffi (default)PosixPtySystem✅ Shipping (Linux + macOS)
autoSame as unset✅ Equivalent to posix-ffi
sidecar—❌ Throws UnsupportedPlatformException (deferred to phase 12)
pecl—❌ Throws UnsupportedPlatformException (deferred to phase 12)

When implementing a new backend:

  1. Implement Contract\PtySystem (and the full contract tree).
  2. Add the dispatch case in PtySystemFactory::default().
  3. Wire the env-var-to-backend mapping in the same method.
  4. Remove the UnsupportedPlatformException::forDeferredBackend() throw for that value.

ChildPollTrait — child lifecycle fast-path

The ChildPollTrait (in src/Posix/ChildPollTrait.php) is a package-private trait shared by PosixChild and PosixProcess. It exposes the canonical lifecycle methods: pid(), exited(), wait(), exitCode(), and pollDestruct().

Exit detection uses a two-tier strategy:

  1. Fast path (sub-ms): tryWaitpid(int $pid): ?int calls waitpid(pid, &status, WNOHANG) via FFI into libc. Returns the exit code immediately if the child has exited, null if still running or FFI is unavailable. WNOHANG makes it non-blocking — returns 0 if the child is still alive.
  2. Fallback (10 ms poll): PHP's proc_get_status() polled in a usleep(10_000) loop inside wait(), or a single probe in exited().

Exit code decoding: normally-exited → (status >> 8) & 0xFF; signal-terminated → 128 + (status & 0x7F) (Unix shell convention). The FFI call is wrapped in try/catch — any FFI\Exception or PtyException falls through to the poll loop silently.

The tryWaitpid method is private — it is not part of the public API. Implementors of custom Child backends do not need to replicate it unless they want FFI-based fast-path exit detection.

Internal PTY allocation — openpty vs quartet

The PosixPtySystem backend uses two platform-specific code paths internally when allocating the master fd:

  • Darwin (macOS) — openpty(3) is attempted first (single libc call opening both master and slave). If it returns -1, falls back to the POSIX quartet: posix_openpt + grantpt + unlockpt + ptsname_r. The slave fd returned by openpty() is closed immediately — the child gets its stdio via proc_open.
  • Linux — The POSIX quartet is used directly. The openpty(3) symbol on Linux lives in libutil.so.1 rather than libc; Libc::libutil() loads it as a separate FFI handle specifically for this symbol. Most Linux systems have both, so the Darwin-first path is not normally exercised on Linux.

This internal detail is documented here for developers extending PosixPtySystem or implementing a parallel backend. Consumer code is fully abstracted by Pty::open() and the PtySystem interface.

Testing your extension

Use a FakePtySystem that implements PtySystem and returns synthetic PtyPair / Child objects. This lets unit tests run without FFI or real syscalls.

use SugarCraft\Pty\Contract\PtySystem;

final class FakePtySystem implements PtySystem
{
    public function open(int $cols, int $rows): PtyPair { /* ... */ }
}

// In tests:
$system = new FakePtySystem();
$pair = $system->open(80, 24);

For integration tests that need real PTY behaviour (resize forwarding, SIGWINCH, controlling terminal), use the real PosixPtySystem on CI. Darwin/macOS tests run in a separate CI job (see .github/workflows/ci.yml macos-test matrix).

Versioning policy

  • @api — The Contract\* interfaces are stable. Implementing them is the supported extension path.
  • @internal — All Posix\* classes, PtySystemFactory, and SignalForwarder are internal. Do not extend or call them directly from outside the lib.
  • Backend selection — The env var dispatch in PtySystemFactory::default() is part of the stable API. Adding new backend values is non-breaking as long as unknown values throw InvalidArgumentException.
