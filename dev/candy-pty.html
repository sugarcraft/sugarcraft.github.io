Extending and testing the PTY backend

Extension points

The primary extension point is the Contract\PtySystem interface. Implement this interface to provide a custom PTY backend (e.g. Windows ConPTY sidecar, pecl extension):

interface PtySystem
{
    public function open(int $cols, int $rows): PtyPair;
}

Where PtyPair, MasterPty, SlavePty, and Child are also contracts in Contract/. See candy-pty/src/Contract/ for the full namespace tree.

Backend conventions

All backends must be selectable via SUGARCRAFT_PTY_BACKEND. The factory PtySystemFactory::default() reads this env var and dispatches:

ValueBackend classStatus
posix-ffi (default)PosixPtySystem✅ Shipping (Linux + macOS)
autoSame as unset✅ Equivalent to posix-ffi
sidecar—❌ Throws UnsupportedPlatformException (deferred to phase 12)
pecl—❌ Throws UnsupportedPlatformException (deferred to phase 12)

When implementing a new backend:

  1. Implement Contract\PtySystem (and the full contract tree).
  2. Add the dispatch case in PtySystemFactory::default().
  3. Wire the env-var-to-backend mapping in the same method.
  4. Remove the UnsupportedPlatformException::forDeferredBackend() throw for that value.

Testing your extension

Use a FakePtySystem that implements PtySystem and returns synthetic PtyPair / Child objects. This lets unit tests run without FFI or real syscalls.

use SugarCraft\Pty\Contract\PtySystem;

final class FakePtySystem implements PtySystem
{
    public function open(int $cols, int $rows): PtyPair { /* ... */ }
}

// In tests:
$system = new FakePtySystem();
$pair = $system->open(80, 24);

For integration tests that need real PTY behaviour (resize forwarding, SIGWINCH, controlling terminal), use the real PosixPtySystem on CI. Darwin/macOS tests run in a separate CI job (see .github/workflows/ci.yml macos-test matrix).

Versioning policy

  • @api — The Contract\* interfaces are stable. Implementing them is the supported extension path.
  • @internal — All Posix\* classes, PtySystemFactory, and SignalForwarder are internal. Do not extend or call them directly from outside the lib.
  • Backend selection — The env var dispatch in PtySystemFactory::default() is part of the stable API. Adding new backend values is non-breaking as long as unknown values throw InvalidArgumentException.

Namespace tree

SugarCraft\Pty
├── Contract           # Stable API interfaces
│   ├── PtySystem      # open(cols, rows): PtyPair
│   ├── PtyPair        # master(): MasterPty, slave(): SlavePty
│   ├── MasterPty      # read/write/resize/size/isClosed/stream/close
│   ├── SlavePty        # path/spawn
│   ├── Child          # pid/exited/wait/exitCode/kill
│   └── Pump           # run(master, stdin, stdout, child?, opts?): int
├── Exception
│   └── UnsupportedPlatformException
├── Posix              # Internal POSIX/FFI implementation
│   ├── PosixPtySystem
│   ├── PosixPtyPair
│   ├── PosixMasterPty
│   ├── PosixSlavePty
│   ├── PosixChild
│   └── PosixPump
├── PtySystemFactory   # Entry point; SUGARCRAFT_PTY_BACKEND dispatch
├── SignalForwarder    # SIGWINCH → resize, SIGCHLD → reap
└── PumpOptions        # Callback holders + sshDefault()
