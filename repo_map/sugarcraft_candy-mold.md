# SugarCraft/candy-mold — Innovation & Comparison Report

## Overview

**candy-mold** is the `composer create-project` bootstrap scaffold for the SugarCraft monorepo — a PHP port of the Go [charmbracelet/bubbletea-app-template](https://github.com/charmbracelet/bubbletea-app-template) ecosystem. It is **not a library** — it is a starter template that developers fork to bootstrap new SugarCraft TUI applications. It ships a runnable counter `Model` wired through `Program`, demonstrating every architectural concept a SugarCraft developer needs to understand.

**Key statistics:**
- **5 files** in the scaffold (`bin/start`, `src/Counter.php`, `tests/CounterTest.php`, `composer.json`, `phpunit.xml`)
- **1 demo Model** (`Counter`) implementing the full TEA triad
- **0 external library deps beyond core** — only `candy-core` + `candy-sprinkles`
- **Status**: v1 ready (🟢 in MATCHUPS.md)

---

## What Makes candy-mold Unique

Unlike every other SugarCraft library (which are composable leaf packages), candy-mold is a **project scaffold**. Its job is not to provide reusable components — it is to show a developer how to structure a complete, runnable TUI app. The README's own words:

> "Skeleton repo for bootstrapping a SugarCraft TUI app. Pour your model into the mold and you've got a working app."

This is directly analogous to `charmbracelet/bubbletea-app-template` for Go — a starter that shows `tea.Model`/`tea.Msg`/`tea.Update`/`tea.View` in action, not a library you'd import into an existing project.

## Scaffold Architecture

### File Layout

```
my-app/                        ← generated by `composer create-project sugarcraft/candy-mold my-app`
├── composer.json              # requires candy-core + candy-sprinkles
├── phpunit.xml                 # bootstrap="vendor/autoload.php", colors, .phpunit.cache/
├── bin/start                   # entry point — 3 meaningful lines of code
├── src/
│   └── Counter.php            # demo Model (72 lines, full docblock explaining TEA)
└── tests/
    └── CounterTest.php        # 7 deterministic unit tests
```

### Entry Point: `bin/start`

```php
#!/usr/bin/env php
<?php
// ...
require $path;  // locate vendor/autoload.php

use App\Counter;
use SugarCraft\Core\Program;

(new Program(new Counter()))->run();
```

**Exactly 3 meaningful lines** (excluding boilerplate path-searching and panic handler comments). The `Program` harness owns:
- Event loop (ReactPHP `addReadStream`/`addPeriodicTimer`)
- Raw mode setup / alt screen lifecycle
- Signal handling (SIGINT/SIGWINCH/SIGTSTP)
- Frame rendering tick + diffing
- Input parsing via `InputReader`

The developer writes only the `Model`. This is the same design philosophy as bubbletea's `tea.NewProgram(initialModel()).Run()` — the framework handles all terminal plumbing.

### The Counter Model: `src/Counter.php`

```php
final class Counter implements Model
{
    public function __construct(public readonly int $n = 0) {}

    public function init(): ?\Closure          { return null; }

    public function update(Msg $msg): array
    {
        if (!$msg instanceof KeyMsg) {
            return [$this, null];
        }
        if ($msg->type === KeyType::Char && $msg->rune === 'q') {
            return [$this, Cmd::quit()];
        }
        return [match ($msg->type) {
            KeyType::Up   => new self($this->n + 1),
            KeyType::Down => new self($this->n - 1),
            default       => $this,
        }, null];
    }

    public function view(): string
    {
        $body = sprintf("  count: %d  \n  ↑ ↓ to change · q to quit  ", $this->n);
        return Style::new()
            ->border(Border::rounded())
            ->padding(1, 2)
            ->render($body);
    }

    public function subscriptions(): ?Subscriptions { return null; }
}
```

**Immutability**: The `readonly int $n` is the only state. Every state transition returns a **new** `Counter` instance rather than mutating `$this`. This is the Elm architecture principle: state lives in value objects, transitions are pure functions.

**TEA Pattern Demonstration:**
1. **`init()`** — Returns `null` (no startup command). A real app might return `Cmd::tick(1.0, fn() => new TickMsg())` here for a clock.
2. **`update(Msg $msg): array`** — Pure state transition. `[$nextModel, $nextCmd] = $counter->update($keyMsg)`. The array destructuring is how PHP simulates Go tuples.
3. **`view(): string`** — Declarative rendering. Returns ANSI string with rounded border + padding via `candy-sprinkles`.
4. **`subscriptions()`** — Returns `null` (no background subscriptions). A real app might return tick timers here.

### Test Suite: `tests/CounterTest.php`

```php
final class CounterTest extends TestCase
{
    public function testUpIncrementsCount(): void
    {
        [$next, $cmd] = (new Counter())->update(new KeyMsg(KeyType::Up, ''));
        $this->assertInstanceOf(Counter::class, $next);
        $this->assertSame(1, $next->n);
        $this->assertNull($cmd);
    }

    public function testQuitDispatchesQuitCmd(): void
    {
        [$next, $cmd] = (new Counter(7))->update(new KeyMsg(KeyType::Char, 'q'));
        $this->assertNotNull($cmd, 'quit returns Cmd::quit()');
    }

    public function testUpdateIsPure(): void
    {
        $start = new Counter(10);
        $start->update(new KeyMsg(KeyType::Up, ''));
        $this->assertSame(10, $start->n, 'original Counter must remain unchanged');
    }
}
```

**7 tests total:**
1. `testStartsAtZero` — default state
2. `testUpIncrementsCount` — KeyMsg → new Counter
3. `testDownDecrementsCount` — KeyMsg with initial value
4. `testQuitDispatchesQuitCmd` — `q` key → `Cmd::quit()`
5. `testNonKeyMessageIgnored` — `WindowSizeMsg` passes through unchanged
6. `testInitReturnsNoCmd` — `init()` returns null
7. `testViewContainsCount` — `view()` includes count + help text
8. `testUpdateIsPure` — immutability check

**Key insight**: No event loop, no terminal, no mocking. Direct `->update($msg)` calls asserting the returned `[Model, ?Cmd]` tuple. This is the testing pattern for Elm-architecture models — deterministic, synchronous unit tests.

## TEA Pattern Demonstration

The counter demonstrates **The Elm Architecture** (also called The TEA Pattern):

```
┌─────────────────────────────────────────────────────────────┐
│                         Program                             │
│  ┌─────────────┐   ┌──────────────┐   ┌──────────────────┐  │
│  │  stdin      │──▶│ InputReader │──▶│ dispatch(Msg)    │  │
│  │  (TTY bytes) │   │  (parse)     │   │                  │  │
│  └─────────────┘   └──────────────┘   │  ┌───────────┐  │  │
│                                        │  │  update()  │──┼──┼──▶ [new Model, ?Cmd]
│                                        │  └───────────┘  │  │
│                                        │                  │  │
│  ┌─────────────┐   ┌──────────────┐   │  ┌───────────┐  │  │
│  │  stdout    │◀──│  Renderer    │◀──│  │   view()  │──┼──┘
│  │  (ANSI)    │   │  (diff+paint)│   │  └───────────┘  │  │
│  └─────────────┘   └──────────────┘   └──────────────────┘  │
└─────────────────────────────────────────────────────────────┘
```

**The Model** (`Counter`) is a value object with:
- `public readonly int $n` — immutable state
- `init(): ?Closure` — startup command (null here)
- `update(Msg $msg): array` — pure transition function
- `view(): string` — renders current state as ANSI string
- `subscriptions(): ?Subscriptions` — background subscriptions (null here)

**The Message** (`KeyMsg`):
- Carries key type (`Up`/`Down`/`Char`) and rune (`'q'`)
- Dispatched by `InputReader` when ↑/↓/q keys are pressed

**The Command** (`Cmd::quit()`):
- A `Closure(): ?Msg` factory for side effects
- `Cmd::quit()` returns a closure that produces `QuitMsg`, signaling program exit
- Other commands (`Cmd::tick()`, `Cmd::batch()`, `Cmd::exec()`) handle timers, concurrency, shell-out

**The View** (`Style::new()->border(Border::rounded())->padding(1, 2)->render($body)`):
- Uses `candy-sprinkles` (port of `lipgloss`) for declarative ANSI styling
- `Border::rounded()` produces `╭──╮│  │╰──╯` style box
- `padding(1, 2)` adds 1 line top/bottom, 2 cols left/right

## CI/CD Integration

candy-mold has **no `.github/workflows/` directory of its own**. Instead, it is tested via the **monorepo-wide `ci.yml`** at the root of `sugarcraft/`. The matrix computation in `scripts/affected-libs.php` discovers `candy-mold` by scanning for `composer.json` + `phpunit.xml` files.

From `scripts/affected-libs.php`, the reverse-dependency graph determines that changing `candy-core` affects every consumer including `candy-mold`. A change to `candy-mold/src/Counter.php` affects only `candy-mold` itself.

**CI test command:**
```bash
cd candy-mold && composer install --prefer-dist --no-progress -n
php candy-mold/vendor/bin/phpunit -c candy-mold/phpunit.xml --colors=never
```

**Test isolation**: The counter tests are fully deterministic — no mocking, no TTY, no event loop. They run in milliseconds and are thread-safe.

- candy-mold demonstrates **immutability** explicitly (`readonly int $n`, `with*()` pattern implied by `new self($this->n + 1)`)

## What candy-mold Gets Right

1. **Minimal surface area** — 5 files, 72-line Model. A developer can read the entire scaffold in 10 minutes.

2. **Self-contained demonstration** — `bin/start` is literally 3 lines. The framework is invisible; only the Model matters.

3. **Deterministic testing** — No event loop, no TTY, no mocking. `CounterTest` shows the correct testing pattern for Elm-architecture: construct a `Msg`, call `update()`, assert the returned tuple.

4. **Docblock-first teaching** — The `Counter.php` docblock (~20 lines) explains Elm architecture, immutability, and the `->with*()` convention explicitly. The README's "Anatomy of a SugarCraft Model" section is a direct tutorial.

5. **`composer create-project` lifecycle** — The scaffold is a proper Composer package (`type: project`, `bin: bin/start`, `autoload: App\\`), installable via `composer create-project`. This is the correct Composer pattern for project templates.

6. **Path-repo closure** — The `composer.json` has `repositories[]` pointing at sibling directories (`../candy-core`, `../candy-sprinkles`). This lets the monorepo development workflow work without publishing to Packagist.

7. **VHS demo** — `.vhs/start.tape` renders a GIF showing the counter being driven with ↑/↓/q. CI renders this to `.vhs/start.gif` via the `vhs.yml` workflow.

## Limitations and Improvement Opportunities

1. **No sub-component composition** — The counter is a single flat Model. The Go template demonstrates `spinner.Model` as a field of the parent `model` struct, showing composition. Adding a `SpinnerModel` sub-component to the counter would be more illustrative.

2. **No `init()` command** — `init()` returns `null`. A real-world scaffold might return `Cmd::tick(1.0, fn() => new TickMsg())` to demonstrate the startup command pattern. This would let developers see the clock tick.

3. **No multiple screens/pages** — All apps start simple. But the scaffold could demonstrate a two-screen app (e.g., a counter that navigates to a "settings" screen) using `ScreenStack`. The current scaffold gives no hint about how to do multi-screen.

4. **No error handling** — The Go template has a custom `errMsg error` type for typed error routing. The counter has no error path. Adding a `LoadConfigCmd` that could fail would demonstrate error handling.

5. **No external I/O demonstration** — `Cmd::exec()`, `Cmd::promise()`, `Cmd::http()` are not shown. A version that shells out to `git status` or fetches an API would demonstrate async commands.

6. **No `Cmd::batch()`/`Cmd::sequence()`** — The counter's quit command returns `Cmd::quit()` directly. Showing a version that uses `Cmd::batch()` to run multiple concurrent commands would demonstrate the concurrency model.

7. **Bin entry point is minimal** — The Go template's `main.go` includes explicit error handling with `fmt.Println(err); os.Exit(1)`. candy-mold's `bin/start` delegates all error handling to `Program::run()` silently.

8. **No integration with sugar-bits components** — The README's "common next steps" table mentions `sugar-bits` Spinner, `sugar-prompt` Group, etc., but the scaffold itself uses only raw `Style` from `candy-sprinkles`. A version that embeds a `Spinner` from `sugar-bits` would show the component model.

9. **Tests do not use data providers** — The 7 tests are explicit individual methods. Using `@dataProvider` for the key-type → expected-count matrix would reduce boilerplate.

10. **CI is monorepo-only** — Since candy-mold has no own `.github/workflows/`, it can only be tested within the monorepo. A standalone `composer create-project` user would need to set up their own CI.

## Architectural Insights

### The Immutability Pattern

The counter uses PHP 8.3 `readonly` properties:

```php
public function __construct(public readonly int $n = 0) {}
```

This forces **nominal immutability** — the property cannot be reassigned after construction. Combined with `new self($this->n + 1)` in `update()`, this ensures the Elm guarantee: `update()` always returns a new Model, never mutates the old one.

The Go template does not enforce immutability — it uses a struct with mutable fields:
```go
type model struct {
    spinner  spinner.Model
    quitting bool
    err      error
}
```

The Go counter mutates `m.quitting = true` and returns the same struct instance. This works in Go because the TEA loop doesn't hold references to the old model after `Update()` returns. In PHP, `readonly` prevents accidental mutation that would violate this contract.

### The Tuple Simulation

Go's `Update` returns a typed tuple:
```go
func (m model) Update(msg tea.Msg) (tea.Model, tea.Cmd)
```

PHP has no native tuples. The convention is an array destructure:
```php
[$next, $cmd] = $counter->update($msg);
```

This is convention-only — PHP treats it as a general `array`. There is no compile-time enforcement that the first element is a `Model` and the second is `?Closure`. The `CounterTest` demonstrates this pattern correctly but relies on developer discipline.

### The `?Closure` Return Type for Commands

`Cmd::quit()` returns `Closure`: `fn() => new QuitMsg()`. This is a deferred computation — the command is **scheduled** by the runtime, not executed immediately. The command factory produces a closure that the event loop calls on the next tick.

```php
if ($msg->type === KeyType::Char && $msg->rune === 'q') {
    return [$this, Cmd::quit()];
}
```

`Cmd::quit()` is:
```php
public static function quit(): \Closure
{
    return fn (): ?Msg => new QuitMsg();
}
```

This is the PHP equivalent of Go's `tea.Cmd`:
```go
func Quit() Cmd {
    return func() Msg {
        return QuitMsg{}
    }
}
```

## Notable Clever Implementation Details

### 1. The `subscriptions()` null-default

`Counter::subscriptions()` returns `null`. The `Model` interface requires this method. The `SubscriptionCapable` trait in `candy-core` provides a default implementation returning `null`, so implementing classes don't need to declare it. Since `Counter` doesn't use subscriptions, this is the right trade-off.

### 2. The `Style::new()->border()->padding()->render()` fluent chain

`candy-sprinkles` uses fluent builders:

```php
return Style::new()
    ->border(Border::rounded())
    ->padding(1, 2)
    ->render($body);
```

Each `with*()` method returns a **new `Style`** instance (via private `mutate()` helper), so the chain is immutable. This mirrors the same pattern in `lipgloss`.

### 3. The `bin/start` autoload search

```php
foreach (['vendor/autoload.php', __DIR__ . '/../vendor/autoload.php'] as $candidate) {
    if (is_file($path)) { require $path; return; }
}
```

This searches two locations (sibling `vendor/` and parent `vendor/`) so the script works both:
- When invoked as `./bin/start` from the project root (autoload is at `./vendor/autoload.php`)
- When invoked as `php bin/start` from the project root (path is the same)

### 4. The optional panic handler

```php
// use SugarCraft\Log\Log;
// Log::installPanicHandler();
```

The `bin/start` comments out the panic handler, making it opt-in. This keeps the default experience clean while documenting that `candy-log` support exists.

## Conclusion

candy-mold is a **minimal, well-designed scaffold** that correctly demonstrates the Elm Architecture in PHP. Its 72-line `Counter` Model shows immutability, message dispatch, command scheduling, and declarative styling. The 7-unit test suite demonstrates deterministic model testing without event loops or mocking. The `composer create-project` lifecycle makes it the correct entry point for PHP developers new to SugarCraft.

The most significant improvement opportunity is **adding sub-component composition** (a `Spinner` from `sugar-bits` as a field of `Counter`) to mirror the Go template's demonstration of `spinner.Model` within the parent model. This would show developers how to build larger apps from smaller TEA components.

As a scaffold, candy-mold succeeds at its job: **get a developer from zero to a running SugarCraft TUI app in under a minute**. The architecture is clean, the documentation is clear, and the test coverage is meaningful. It is a solid foundation for the SugarCraft ecosystem.
